# Milestone 5: First Fireworks RFT Run

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Run the first reinforcement fine-tuning job on `accounts/fireworks/models/qwen3-8b` using the Fireworks platform, producing a tuned model that we can compare against the prompt-only baseline.

**Architecture:** We use the `eval-protocol` CLI to orchestrate the end-to-end workflow: it packages and deploys the evaluator, uploads the dataset, and launches the RFT job — all in a single command. The existing evaluator code (`evaluators/shell_rft_v0/`) and training data (`data/train.jsonl`) are already complete from milestones 2–4. This milestone is primarily operational — installing tools, verifying the evaluator deploys correctly, launching the job, then running the tuned model through the same baseline script to measure improvement.

**Tech Stack:** `eval-protocol` CLI, `reward-kit` (already installed via `fireworks-ai[reward-kit]`), Fireworks platform, existing evaluator and dataset.

## Design Decisions

### eval-protocol vs. manual firectl

The design doc specifies `eval-protocol create rft` as the path. This is the right choice: it handles dataset upload, evaluator deployment, and job creation in a single orchestrated command. The alternative (`reward-kit deploy` + `firectl rftj create`) requires manually tracking resource names across steps.

### RFT parameters — start with defaults

Fireworks recommends starting with defaults. For the first run:

- **n=4** rollouts per prompt (Fireworks default, minimum 2)
- **epochs=1** (single pass to see if learning signal exists)
- **learning_rate**: default (Fireworks recommends letting the platform pick)
- **lora_rank**: default (8)
- **temperature=0.7** for rollouts (Fireworks default; higher than baseline's 0.0 to encourage exploration)
- **max_tokens=256** for rollouts (these are single shell commands, not reasoning chains — Fireworks handles the rollout generation so we don't need room for `<think>` tokens)

We will NOT tweak these until we have one complete run to baseline against.

### Validation dataset

We'll pass `data/val.jsonl` as the evaluation dataset so Fireworks can track held-out reward during training. The test set (`data/test.jsonl`) stays reserved for our final comparison.

### Post-training evaluation

After the job completes, we run the exact same `scripts/run_local_baseline.py` against the tuned model to get a head-to-head comparison on `data/test.jsonl`. Same evaluator pipeline, same metrics, different model ID.

### Qwen3 reasoning tokens at inference

The tuned model will likely still emit `<think>` blocks. Our baseline script already strips these (added in milestone 4). We'll use the same approach when evaluating the tuned model.

Note on max_tokens: for RFT rollouts, 256 tokens is fine because Fireworks generates rollouts and the evaluator scores the final command. For post-training *inference* evaluation, we use 4096 to give the model room for its reasoning chain before the command.

### Baseline context

The prompt-only baseline achieves 82% exact match overall (96% file_counting, 88% topk_by_size, 76% content_search, 68% csv_filtering). This is near the "too easy" threshold we set in milestone 4, but per-family variance is significant — csv_filtering and content_search have clear room for improvement. A meaningful result would be pushing overall accuracy above 88% or showing >10pp gains on the weaker families. Even a modest improvement validates the end-to-end workflow, which is the primary goal of this milestone.

```python
"""Compare baseline and RFT results side by side."""

import json
import sys


def main():
    baseline_path = sys.argv[1] if len(sys.argv) > 1 else "data/baseline_results.json"
    rft_path = sys.argv[2] if len(sys.argv) > 2 else "data/rft_results.json"

    with open(baseline_path) as f:
        baseline = json.load(f)
    with open(rft_path) as f:
        rft = json.load(f)

    bm = baseline["metrics"]
    rm = rft["metrics"]

    print("=" * 60)
    print("HEAD-TO-HEAD COMPARISON")
    print(f"  Baseline model: {baseline['model']}")
    print(f"  RFT model:      {rft['model']}")
    print("=" * 60)
    print(f"{'Metric':<25} {'Baseline':>10} {'RFT':>10} {'Delta':>10}")
    print(f"{'-'*25} {'-'*10} {'-'*10} {'-'*10}")

    for key in ["exact_match_rate", "mean_reward", "parse_success_rate"]:
        bv = bm[key]
        rv = rm[key]
        delta = rv - bv
        sign = "+" if delta > 0 else ""
        print(f"{key:<25} {bv:>10.1%} {rv:>10.1%} {sign}{delta:>9.1%}")

    print()
    print("Per task family:")
    print(f"{'Family':<20} {'Baseline':>10} {'RFT':>10} {'Delta':>10}")
    print(f"{'-'*20} {'-'*10} {'-'*10} {'-'*10}")
    for family in sorted(bm["by_family"].keys()):
        bv = bm["by_family"][family]["exact_match_rate"]
        rv = rm["by_family"].get(family, {}).get("exact_match_rate", 0.0)
        delta = rv - bv
        sign = "+" if delta > 0 else ""
        print(f"{family:<20} {bv:>10.1%} {rv:>10.1%} {sign}{delta:>9.1%}")
    print("=" * 60)


if __name__ == "__main__":
    main()
```

## Notes

### Timing

Task 3 launches an async training job. There will be a gap (potentially hours) between Task 3 and Task 4 while the job runs. This is normal.

### Cost

Fireworks' free-tuning threshold covers models under 16B parameters, and Qwen3-8B qualifies. The primary cost is GPU time for rollout generation and training. For 500 examples with n=4 rollouts, this should be modest.

### Fallback: manual workflow

If `eval-protocol create rft` doesn't work as expected (tool is evolving), the manual path is:

1. Deploy evaluator: `reward-kit deploy --id shell-rft-v0 --metrics-folders "shell_rft=./evaluators/shell_rft_v0" --force`
2. Upload dataset via Fireworks UI or API
3. Create RFT job via Fireworks UI or `firectl rftj create`

The evaluator code and dataset are identical regardless of which tool orchestrates the upload.

### What this milestone does NOT include

- **Hyperparameter tuning**: We use defaults for the first run. Tuning is a post-milestone-5 activity.
- **Multiple runs**: One run to establish the workflow and get initial results.
- **Model deployment for production**: We only need inference access to compare against the baseline.
- **Baseline B (grammar-constrained)**: Deferred — the prompt-only baseline already established that parse failures are <2% of errors.
